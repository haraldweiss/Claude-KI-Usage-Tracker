# Design Spec: Multi-User Auth (Variante Y — getrennte Daten)

**Date:** 2026-04-29
**Project:** Claude Usage Tracker
**Status:** Draft — to implement in a separate session, replaces HTTP Basic Auth on the VPS

---

## Why this exists

The VPS currently serves the dashboard at `https://wolfinisoftware.de/claudetracker/`
behind Apache HTTP Basic Auth. Two limitations:

1. **UX rough edge:** the browser drops the cached credentials when navigating
   between sub-routes (e.g. Settings tab) and re-prompts. Annoying once a day.
2. **Single-user only:** one shared password, one shared dataset. The user's
   stated goal is to share the tool with friends/family in the future, with
   each person seeing only their own claude.ai/API-Key data.

This spec replaces Basic Auth with a **proper login system** that supports
multiple users out of the box, while keeping the implementation pragmatic.

The chosen variant is **Y**: data is partitioned per user. Every user logs in,
sees only their own usage, has their own extension config. The admin (Harald)
can enable/disable accounts but **cannot see other users' costs**.

## Architecture

```
┌─────────────────────────────────────────────────────────────────┐
│ Apache vhost — wolfinisoftware.de/claudetracker/                │
│ (HTTPS terminates here, no auth at Apache layer anymore)        │
└──────────────┬─────────────────────────────────────┬────────────┘
               │                                     │
               │ /claudetracker/* (static)           │ /claudetracker/api/*
               ▼                                     ▼
        Frontend (Vite SPA)                  Node backend (Express)
        - /login route                       - JWT-based auth middleware
        - /admin route (admin only)          - User table in SQLite
        - /dashboard, /models, /combined     - Per-row user_id on usage_records
        - All API calls send                 - /api/auth/login → JWT
          Authorization: Bearer <jwt>        - /api/auth/me → current user
                                             - /api/admin/users (admin only)
                                             - /api/admin/invite (admin only)
                                             - All data queries scope by user_id
```

**Replaces:** HTTP Basic Auth in `/etc/httpd/conf.d/claudetracker.conf` is removed.
The Apache vhost just proxies to Node again — auth is enforced inside Node.

## Database

Three table changes, all additive via `addMissingColumns`:

### New table: `users`

```sql
CREATE TABLE users (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  username TEXT NOT NULL UNIQUE,
  password_hash TEXT NOT NULL,
  role TEXT NOT NULL CHECK (role IN ('admin', 'user')),
  enabled INTEGER NOT NULL DEFAULT 1,
  created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
  last_login_at DATETIME
);
```

Bcrypt hashes (cost factor 12). Username is the primary login handle (not email,
to keep things simple — no password reset flow in v1).

### New table: `invites`

```sql
CREATE TABLE invites (
  token TEXT PRIMARY KEY,                 -- 32-byte hex, generated by admin
  created_by_user_id INTEGER NOT NULL REFERENCES users(id),
  username TEXT,                          -- pre-set username, optional
  expires_at DATETIME NOT NULL,           -- default: now + 7 days
  used_at DATETIME,                       -- set when redeemed
  used_by_user_id INTEGER REFERENCES users(id)
);
```

Magic-link flow: admin generates a token, gives it to the invitee, invitee opens
`https://wolfinisoftware.de/claudetracker/invite/<token>`, sets username +
password, becomes a regular `user` (never `admin`).

### `usage_records` — add `user_id` column

```sql
ALTER TABLE usage_records ADD COLUMN user_id INTEGER REFERENCES users(id);
CREATE INDEX idx_usage_user ON usage_records(user_id);
```

Migration: existing rows get `user_id = 1` (the bootstrap admin = Harald).
Going forward, every `POST /api/usage/track` sets `user_id` from the JWT.

## Backend changes

### Auth foundation

**New service: `authService.ts`**
- `hashPassword(plain) → bcrypt`
- `verifyPassword(plain, hash) → bool`
- `signToken({ id, role }) → JWT` (24h expiry, HS256)
- `verifyToken(jwt) → { id, role } | throws`

JWT secret stored in `JWT_SECRET` env var, loaded by systemd unit. Generate with
`openssl rand -base64 48` once and persist.

**New middleware: `requireAuth`**
- Reads `Authorization: Bearer <token>` header
- Verifies JWT, loads user from DB, checks `enabled = 1`
- Attaches `req.user = { id, role, username }`
- Returns 401 on failure

**New middleware: `requireAdmin`**
- Same as `requireAuth` but additionally requires `role = 'admin'`
- Returns 403 if not admin

### New auth routes

```
POST /api/auth/login          { username, password } → { token, user }
GET  /api/auth/me             (auth) → { id, username, role }
POST /api/auth/redeem-invite  { token, username, password } → { token, user }
```

### New admin routes

```
GET  /api/admin/users         (admin) → [{ id, username, role, enabled, last_login_at }]
PUT  /api/admin/users/:id/enable  (admin) → toggles enabled flag
POST /api/admin/invites       (admin) { username?, expires_in_days? } → { token, expires_at }
GET  /api/admin/invites       (admin) → [active invites]
DELETE /api/admin/invites/:token  (admin) → revoke
```

**Privacy guarantee:** the admin endpoints expose only `enabled`, `username`,
`last_login_at` per user — never cost or sync data. Even the admin cannot read
another user's `usage_records`. Confirmed by the admin route handlers never
querying `usage_records` at all.

### Modified routes — scope every read/write by `req.user.id`

- `POST /api/usage/track`: sets `user_id = req.user.id` on insert
- `GET /api/usage/summary`: `WHERE user_id = ?`
- `GET /api/usage/history`: `WHERE user_id = ?`
- `GET /api/usage/models`: `WHERE user_id = ?`
- `GET /api/usage/console/keys`: `WHERE user_id = ?`
- `GET /api/pricing`: shared across users (no scoping — same model prices for everyone)
- `PUT /api/pricing/:model`: admin-only? Or shared? **Decision needed:** keep
  pricing global for now (single source of truth, less to maintain).

### Bootstrap migration

On first start with the new schema:
1. If `users` is empty, create admin from env: `BOOTSTRAP_ADMIN_USER=harald`,
   `BOOTSTRAP_ADMIN_PASS=<pw from env or randomly generated and printed once>`.
2. Backfill `usage_records.user_id = 1` for all existing rows.

## Frontend changes

### New routes

- `/login` — username/password form, calls `/api/auth/login`, stores JWT in
  `localStorage`, redirects to `/`
- `/invite/:token` — accept invite, set username + password
- `/admin` — admin only: list users (enable/disable), generate invites
- All existing routes (`/`, dashboard tabs) wrapped in an `<AuthGate>` that
  redirects to `/login` on missing/expired token

### Auth state

`useAuth()` hook reads token from `localStorage`, fetches `/api/auth/me`, exposes
`{ user, isLoading, login, logout }`.

`api.ts` interceptor adds `Authorization: Bearer <token>` to every request and
catches 401 to force logout.

### UI changes

- Header gets a user menu (top right): username, "Logout", "Admin" link if
  applicable
- Settings page tab gains a "Mein Account" section: change password, see
  last_login_at
- Admin route is a small page with a user table (enable/disable toggles) and
  an invite form (generates a magic-link URL the admin copies and shares
  out-of-band — email, Signal, paper note, whatever)

## Extension changes

### Settings panel additions

In `popup.html`'s `⚙️ Verbindung` section, **replace** the Basic-Auth user/pass
fields with:

- "Login User"
- "Login Passwort"
- A "Anmelden" button that calls `/api/auth/login`, stores the resulting JWT
  in `chrome.storage.local.auth_token`
- Display "✓ Eingeloggt als <username>" after success
- "Abmelden" button removes the token

### `background.js` changes

- `getAuthHeaders()` now reads `auth_token` and returns `Authorization: Bearer <jwt>`
- On 401 response, clear token from storage and surface a notification ("bitte
  erneut anmelden in der Extension")
- Optional: refresh the token when it's about to expire (check `exp` in JWT,
  re-login if < 1h left)

### Migration for existing users

When the new extension version installs:
- If `auth_user` and `auth_pass` are present in storage (Basic Auth from before),
  call `/api/auth/login` once to upgrade them to a JWT, then delete the basic
  creds. One-shot transition, transparent to the user.

## Implementation phases

Each phase is independently deployable.

### Phase 1: Backend auth foundation
- Migration: `users`, `invites` tables, `user_id` column.
- `authService`, `requireAuth`, `requireAdmin` middleware.
- `/api/auth/login` and `/api/auth/me` only.
- Bootstrap admin from env on first run.
- All `usage_records` reads/writes scoped by `req.user.id`.
- Tests: login flow, scoping, admin role check.

**Deliverable:** backend rejects unauthenticated requests but no UI yet —
test with curl + manual JWT.

### Phase 2: Frontend login + AuthGate
- `/login` page with form.
- `AuthGate` wraps all routes.
- `api.ts` sends Authorization header from `localStorage`.
- Header user menu with Logout.
- Apache: remove Basic Auth, redeploy.

**Deliverable:** the user (Harald) logs in once via the form and uses the
dashboard normally. No Basic Auth re-prompts.

### Phase 3: Admin + invites
- `/api/admin/users`, `/api/admin/invites`, `/api/admin/invites/:token`.
- `/admin` page with user table and invite generator.
- `/invite/:token` redemption page.

**Deliverable:** Harald can invite a friend by generating a token, sending
them the magic link, they sign up, see only their own (empty) data.

### Phase 4: Extension migration
- Replace Basic-Auth fields with login form + token storage.
- One-shot migration of legacy `auth_user`/`auth_pass`.
- 401 handling.

**Deliverable:** the extension stores a JWT, refreshes on expiry, prompts the
user to re-log in if needed.

## Out of scope for v1

- **Password reset / forgot password.** Admin can manually reset a user's
  password via the admin page (or psql/sqlite directly).
- **Email verification.** Username-only signup, admin vouches via the invite.
- **OAuth / social login.** Username + password is enough for the use case.
- **Per-user pricing overrides.** Pricing stays global.
- **Per-user plan-subscription cost.** The PLAN_SUBSCRIPTION_EUR lookup stays
  global; if friend X has a different plan, they edit the constant. Could be
  per-user later if it becomes a real annoyance.
- **Audit log of admin actions.** Nice-to-have, not v1.
- **Rate limiting on /api/auth/login.** Defer until we see actual abuse;
  the dashboard isn't public-facing in any high-volume way.

---

## Success criteria

- Single-user (Harald) flow stays unchanged: login once, then dashboard works
  for the day, including extension syncs.
- Admin can generate an invite link, send it, the invitee redeems and lands
  in their own empty dashboard.
- Two users active at the same time see completely separated data — verified
  by populating both DBs with one sync each and checking that A doesn't see B's
  costs and vice versa.
- Admin sees the user list but not user costs — verified by checking
  `/api/admin/users` response shape excludes any cost field.
- Disabled user cannot login or call any API — verified by `enabled = 0` returns
  401 across all routes.
